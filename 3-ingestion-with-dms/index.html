Ingestion With DMS: Skip DMS Lab

Introduction

📌 What is AWS DMS?

AWS Database Migration Service (AWS DMS) enables you to migrate databases to AWS quickly and securely, with minimal downtime to your applications. The source database remains fully operational throughout the migration process, ensuring business continuity.

🔄 Migration Types Supported

AWS DMS supports both homogeneous and heterogeneous database migrations:

  • Homogeneous Migration (Same database engines)

    • Example: Oracle ➝ Oracle, MySQL ➝ MySQL

  • Heterogeneous Migration (Different database engines)

    • Example: Oracle ➝ Amazon Aurora, SQL Server ➝ Amazon RDS for PostgreSQL

⚡ Continuous Data Replication

With low-latency replication, AWS DMS allows you to:

  • Stream data from multiple sources into Amazon S3 to build a highly available data lake.
  • Consolidate multiple databases into a petabyte-scale data warehouse by streaming data to Amazon Redshift.

🏆 Option 3: Skip DMS Lab

Steps Overview

  1. Open AWS CloudShell
  2. Copy data from the staging Amazon S3 bucket to your own S3 bucket
  3. Verify the data in Amazon S3

About the Lab Setup

  • The source database is an RDS PostgreSQL instance storing ticket sales for sporting events.
  • AWS DMS (Database Migration Service) is typically used to migrate data from RDS to an S3 bucket.
  • Instead of using DMS, you can directly copy the source data to your S3 bucket and proceed with the lab.
  • In this lab, you will: ✅ Copy data from a centralized S3 bucket
    ✅ Use an AWS Glue Crawler to catalog metadata
    ✅ Transform the data with AWS Glue
    ✅ Query & create a View in Athena
    ✅ Build a dashboard with Amazon QuickSight

Step 1: Open AWS CloudShell

  • Launch AWS CloudShell in your AWS account.
  • If a pop-up appears, close it.
  • AWS CloudShell is region-specific, check AWS Regional Services List for availability.

Step 2: Copy Data from the Staging S3 Bucket

Run the following command in CloudShell, replacing <YourBucketName> with your actual S3 bucket name:

export dmslabs3bucket=<YourBucketName>
aws s3 cp s3://ws-assets-prod-iad-r-iad-ed304a55c2ca1aee/976050cc-0606-4b23-b49f-ca7b8ac4b153/v1/data/pg-source.zip /tmp &&
unzip /tmp/pg-source.zip -d /tmp/ &&
aws s3 cp --recursive --copy-props none /tmp/dms_sample s3://$dmslabs3bucket/tickets/dms_sample

Verify the Data in S3

1. Access the S3 Console

  • Navigate to the AWS S3 Console and locate the data copied through CloudShell terminal.

2. Find Your S3 Bucket

  • Your S3 bucket name follows this format: 
    BucketName/bucket_folder_name/schema_name/table_name/objects/
  • Example from our lab: 
    /<BucketName>/tickets/dms_sample
  • Each table_name has its own separate path.

3. Review a File Using S3 Select

  • Navigate to the player directory.
  • Select the checkbox next to a file name.
  • Click the Actions dropdown button and choose Query with S3 Select.

  • On the Query with S3 Select page:
    • Leave the default values for Input Settings and SQL Query.
    • Click Run SQL query.
